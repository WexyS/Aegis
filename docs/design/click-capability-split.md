# Click Capability Split Design

**Goal:** Clarify click architecture before implementing ClickTool evidence.

**Decision:** Use separate internal capabilities: `browser_click` and `desktop_click`.

The existing natural-language surface may still accept "click", but the planner/parser must normalize it into one of these explicit capabilities, or return `clarification_required`. Browser and desktop clicks have different evidence, risk, approval, and future integration paths. Keeping them separate prevents one generic click verifier from producing misleading "verified" results.

---

## Chosen Architecture

### Recommendation

Choose Option A:

- `browser_click`
- `desktop_click`

Do not use `click` with `mode="browser" | "desktop"` as the core internal contract.

### Why

**Evidence semantics:** Browser click can rely on browser/page/DOM-like evidence. Desktop click must rely on foreground HWND, PID, window rect, coordinate geometry, DPI/monitor context, and later vision/accessibility. These are not the same proof model.

**Risk classification:** Browser click is usually scoped to the controlled browser page. Desktop click can affect any foreground OS surface and should be treated as higher risk by default.

**Approval handling:** Browser click may be auto-allowed for low-risk deterministic DOM targets. Desktop click should require stronger target evidence and may require approval when the target cannot be identified.

**Parser/decomposition:** Natural language needs routing. "Click the Save button on this webpage" and "click at 500,300" should not enter the same action path.

**Target resolution:** Users should usually express semantic targets, not coordinates. Runtime should resolve "Save button", "first search result", or "Chrome search box" into a browser or desktop target before any click dispatch.

**Future vision/accessibility integration:** Browser click can later upgrade through DOM selectors and accessibility roles. Desktop click can later upgrade through OS accessibility, semantic screen anchors, and vision. These should remain separate adapters feeding the same verification principles.

---

## Target Resolution Layer

Click should be a two-stage operation:

```text
user intent
-> target resolution
-> browser_click / desktop_click / clarification_required
-> evidence capture
-> dispatch only if safe enough
```

The user should not normally provide raw coordinates. Coordinates are a fallback and low-level dispatch primitive, not the main product behavior.

### Browser Target Resolution

Browser clicks should prefer semantic page targets:

- DOM selector
- text match
- role/name
- link/button detection
- input detection
- provider-specific known targets, such as first search result

Examples:

| User request | Target resolution output |
|---|---|
| "click the Save button on this webpage" | `browser_click(target_type="button", name="Save")` |
| "click the first search result" | `browser_click(target="first_search_result")` |
| "click the search box in Chrome" | `browser_click(target_type="input", name_or_role="search")` |

The browser resolver may later emit a selector or coordinate for dispatch, but that selector/coordinate is generated evidence-backed runtime detail, not the user-facing intent.

### Desktop Target Resolution

Desktop clicks need stronger target resolution before dispatch:

- foreground window identity
- process/window title context
- Windows UI Automation, when available
- accessibility tree, later
- OCR, later
- vision anchors, later

Example:

```text
"click the close button in Notepad"
-> resolve foreground/window target
-> resolve close button via UIA/vision/accessibility
-> derive coordinate
-> desktop_click low-level dispatch
```

The coordinate is the last mile, generated by runtime after target resolution. If target resolution is not confident, Aegis should ask for clarification or require approval instead of clicking.

### Resolution Outcomes

| Outcome | Meaning |
|---|---|
| `browser_click` | Browser/page target resolved with enough context for browser evidence. |
| `desktop_click` | Desktop/window target resolved with enough foreground/window evidence. |
| `clarification_required` | Intent is ambiguous or target cannot be identified safely. |
| `approval_required` | Target is identifiable but risk/user-intervention requires confirmation. |

Rule: resolve first, click second. If target cannot be resolved, do not click blindly.

## Evidence Semantics Table

| Capability | Verified Means | Unverified Means | Approval Required Means | Failed Means |
|---|---|---|---|---|
| `browser_click` | A deterministic browser target was observed and click dispatch completed under observable browser context. | Click may have happened, but target/context evidence was missing, ambiguous, or inconsistent. | Browser challenge, permission prompt, or risky page state needs user intervention. | Tool dispatch raised or browser click hard-failed. |
| `desktop_click` | A coordinate click was dispatched inside a known foreground window with PID/HWND/window geometry evidence. | Click may have happened, but target/window/coordinate evidence was missing, ambiguous, or inconsistent. | Target is high-risk, unidentified, or needs human confirmation. | OS click dispatch raised or verifier could not execute. |

Shared rule: dispatch success is never enough for `verified`.

Terminology guardrail: `verified` is a compatibility-level state for the current `ExecutionEvidence` contract. For click work, it must be read as "the declared evidence contract passed", not "the user's semantic intent was definitely achieved".

## Browser Click Evidence Contract

### Can Be Verified Now

- Browser/page context exists before dispatch.
- Page URL before dispatch is observable.
- Selector exists, if selector mode is used.
- Selector resolves to a deterministic target, preferably exactly one visible element.
- Element bounds are available and non-empty.
- Coordinate is inside viewport, if coordinate mode is used.
- Element at point is observable, if coordinate mode is used.
- Dispatch completed without raising.
- Page context is observable after dispatch.
- Bot/CAPTCHA/permission challenge is not detected by existing detection.

### Cannot Be Verified Yet

- The business outcome intended by the user.
- Whether a clicked button semantically meant "Save" unless reliable DOM/accessibility evidence exists.
- Whether a unique selector is semantically correct. `selector_match_count == 1` is deterministic target evidence, not proof that the target means what the user intended.
- Visual-only targets without DOM or accessibility evidence.
- CAPTCHA bypass.
- Full page correctness after click.

### Verified Rule

`browser_click` may be `verified` only if:

- browser/page context exists before dispatch
- final browser context is observable after dispatch
- dispatch completed without raising
- no bot/CAPTCHA/permission challenge is detected
- and one of:
  - selector mode: selector is present, resolves to exactly one visible element, and target bounds are non-empty
  - coordinate mode: x/y are valid integers, viewport is known, coordinate is inside viewport, and element-at-point is observed

### Unverified Rule

Return `unverified` when:

- selector is missing, absent, hidden, or ambiguous
- coordinate is outside viewport
- viewport is unavailable
- element-at-point cannot be observed
- before/after browser context is missing
- target evidence is incomplete

### Approval Required Rule

Return `approval_required` when:

- bot/CAPTCHA challenge is detected
- browser permission prompt blocks the action
- click target is sensitive and needs explicit confirmation
- page state indicates user intervention is required

### Failed Rule

Return `failed` only when:

- click dispatch raises
- browser/page tool hard-fails
- timeout/cancellation aborts execution

### Required Evidence Fields

```python
{
    "tool": "browser_click",
    "mode": "selector" | "coordinates",
    "selector": "button.save" | None,
    "coordinates": {"x": 500, "y": 300} | None,
    "page_url_before": "https://example.test/form",
    "page_url_after": "https://example.test/form",
    "browser_context_observable_before": True,
    "browser_context_observable_after": True,
    "viewport": {"width": 1280, "height": 720, "device_scale_factor": 1},
    "target_before": {"tag": "BUTTON", "role": None, "text_hash": "...", "bounds": {...}} | None,
    "target_after": {"tag": "BUTTON", "role": None, "text_hash": "...", "bounds": {...}} | None,
    "selector_match_count": 1 | None,
    "coordinate_inside_viewport": True | False | None,
    "coordinate_inside_target_bounds": True | False | None,
    "dispatch_ok": True,
    "bot_challenge_detected": False,
    "verification_state": "verified" | "unverified" | "approval_required",
    "verification_reason": "human-readable reason",
    "verification_checks": [
        {"check_name": "...", "expected": "...", "observed": "...", "passed": True, "reason": "..."}
    ],
}
```

Do not store full page content, screenshots, OCR output, or raw sensitive target text. Hash short text evidence if needed.

Important: selector uniqueness is not semantic correctness. A selector can resolve to exactly one visible element and still represent the wrong business action, for example a destructive button with misleading class names. Semantic correctness requires later DOM/accessibility/vision evidence and is out of scope for the first click gate.

### Tests Before Implementation

- selector exact single visible target -> verified/success
- selector missing -> unverified/success false
- selector multiple visible targets -> unverified/success false
- coordinate inside viewport and element observed -> verified/success
- coordinate outside viewport -> unverified/success false
- browser context missing before or after -> unverified/success false
- challenge detected -> approval_required, not verified
- dispatch exception -> failed/success false
- execution evidence contains no full page content

## Desktop Click Evidence Contract

### Can Be Verified Now

- Foreground HWND before dispatch.
- Foreground PID before dispatch.
- Foreground window title/process identity before dispatch.
- Active window rect before dispatch.
- x/y coordinate.
- Coordinate inside active window rect.
- Foreground HWND/PID after dispatch.
- Dispatch completed without raising.

### Cannot Be Verified Yet

- Whether the coordinate corresponds to the user's intended semantic UI control.
- Button label recognition without vision/accessibility.
- UI element identity if only coordinates are available.
- Multi-monitor/DPI correctness unless the runtime captures monitor scale and coordinate space reliably.

### Verified Rule

`desktop_click` may be `verified` only if:

- foreground HWND exists before dispatch
- foreground PID exists before dispatch
- active window rect exists before dispatch
- x/y are valid integers
- coordinate is inside the active window rect in the same coordinate space
- dispatch completed without raising
- foreground state after dispatch is observable
- no unexpected foreground change occurred unless the click target explicitly allows focus/navigation change

Important: coordinate inside active window is geometry verification, not semantic button verification.

Future terminology note: desktop click should eventually expose two layers:

- `geometry_verified`: HWND/PID/window rect/coordinate evidence passed.
- `semantic_verified`: a later vision/accessibility/OS UI target model confirmed the intended control.

Until `semantic_verified` exists, UI copy and reports must avoid implying "the correct button was clicked" for desktop coordinates. A `desktop_click` verified state only means the geometry evidence contract passed.

### Unverified Rule

Return `unverified` when:

- foreground HWND/PID is missing
- active window rect is missing
- x/y are missing or invalid
- coordinate is outside active window rect
- DPI/monitor coordinate space is unknown when required
- foreground changes unexpectedly
- target identity is ambiguous

### Approval Required Rule

Return `approval_required` when:

- target cannot be identified and the click is not explicitly coordinate-only
- target window is high-risk or unknown
- click may affect system/security/payment/destructive UI
- user intent is ambiguous

### Failed Rule

Return `failed` only when:

- OS click dispatch raises
- input subsystem rejects the click
- timeout/cancellation aborts execution

### Required Evidence Fields

```python
{
    "tool": "desktop_click",
    "coordinates": {"x": 500, "y": 300},
    "foreground_before": {"hwnd": 12345, "pid": 1000, "title_hash": "...", "process_name": "app.exe"},
    "foreground_after": {"hwnd": 12345, "pid": 1000, "title_hash": "...", "process_name": "app.exe"},
    "active_window_rect": {"left": 100, "top": 100, "right": 900, "bottom": 700},
    "monitor": {"id": "...", "dpi_scale": 1.0} | None,
    "coordinate_inside_active_window": True,
    "focus_changed": False,
    "dispatch_ok": True,
    "verification_state": "verified" | "unverified" | "approval_required",
    "verification_reason": "human-readable reason",
    "verification_checks": [
        {"check_name": "...", "expected": "...", "observed": "...", "passed": True, "reason": "..."}
    ],
}
```

Do not store screenshots or raw screen text.

### Tests Before Implementation

- coordinate inside known foreground window -> verified/success
- missing foreground HWND -> unverified/success false
- missing active window rect -> unverified/success false
- coordinate outside active window -> unverified/success false
- unexpected foreground change -> unverified/success false
- ambiguous natural-language target -> clarification_required or approval_required, no click dispatch
- dispatch exception -> failed/success false

## Routing Rule

Natural language click intent must route to one of:

- `browser_click`
- `desktop_click`
- `clarification_required`

Examples:

| Input | Route | Reason |
|---|---|---|
| "click the Save button on this webpage" | `browser_click` if browser DOM/context is available | Browser target and page context are implied. |
| "click the first search result" | `browser_click` if browser search page context is available | Browser page semantics are required. |
| "click at 500,300" | `desktop_click` only as explicit low-level fallback | Explicit screen coordinate. |
| "click at 500,300 in the browser" | `browser_click` coordinate mode if browser viewport coordinate space is known, otherwise clarification | Coordinate space matters. |
| "click that button" | `clarification_required` or vision-needed | Target is ambiguous. Do not click blindly. |
| "click OK in the installer window" | `desktop_click` only if foreground/window target evidence is available, otherwise approval/clarification | OS window target. |
| "click Chrome's search box" | target resolution first, then `browser_click` or `desktop_click` depending on resolved context | Semantic target must be resolved before dispatch. |

Routing must happen before dispatch. A generic `click` should not reach execution without a mode.

Critical dependency: click implementation depends on deterministic command decomposition stability. If the planner routes "click the first result in Chrome" to `desktop_click` instead of `browser_click`, the verifier may correctly prove the wrong action. Parser/decomposition must reliably choose `browser_click`, `desktop_click`, or `clarification_required` before click execution is enabled.

Target resolution dependency: `browser_click` and `desktop_click` should generally receive resolved targets, not raw user phrases. A request like "click Save" must go through target resolution before it becomes a selector, role/name, UIA node, or coordinate.

## Risk Table

| Capability | Default Risk | Auto-Allowed When | Approval Needed When |
|---|---|---|---|
| `browser_click` selector | medium | deterministic low-risk page target with single visible match | ambiguous selector, challenge, sensitive page |
| `browser_click` coordinate | medium-high | viewport and element-at-point evidence are strong | target identity unknown |
| `desktop_click` coordinate | high | only explicit coordinate task with strong foreground/window evidence and low-risk context | most natural-language desktop targets |
| `desktop_click` semantic target | high | not auto-allowed initially | requires vision/accessibility/designated target evidence |

## Safety Rules

- Desktop click must never be verified only because dispatch succeeded.
- Browser click must never be verified only because Playwright did not raise.
- Coordinate inside active window is not semantic button verification.
- Coordinate inside browser viewport is not semantic button verification.
- Selector uniqueness is not semantic correctness.
- Click routing must never dispatch when command decomposition is ambiguous.
- Target resolution must happen before click dispatch.
- User-provided coordinates are fallback/low-level primitive input, not the default interaction model.
- Runtime-generated coordinates are acceptable only after semantic target resolution produces enough evidence.
- If target cannot be identified, do not click blindly.
- Vision/accessibility may later upgrade confidence, but they are out of scope now.
- CAPTCHA, bot challenge, permission prompts, and high-risk UI states require approval or user intervention.

## Future Implementation Slices

1. Deterministic command decomposition dependency
   - Design and stabilize routing from natural language to `browser_click`, `desktop_click`, or `clarification_required`.
   - Do not enable click dispatch until ambiguous click language is blocked.

2. Target resolution layer design
   - Design browser target resolution for DOM selector, text, role/name, link/button, input, and provider-specific targets.
   - Design desktop target resolution around foreground window identity, UI Automation, accessibility, OCR, and vision as future adapters.
   - Make `clarification_required` the default for unresolved targets.

3. `browser_click` selector evidence gate
   - Tests first.
   - No desktop click.
   - Verify selector count, visibility, bounds, dispatch, after-context.
   - Do not claim semantic correctness from selector uniqueness alone.

4. `browser_click` coordinate evidence gate
   - Tests first.
   - Verify viewport, coordinate bounds, element-at-point, dispatch, after-context.

5. `desktop_click` coordinate evidence design refinement
   - Confirm coordinate spaces, DPI, monitor handling, foreground window rect APIs.
   - No dispatch until verifier contract is settled.
   - Define how `geometry_verified` is represented without implying `semantic_verified`.

6. `desktop_click` coordinate evidence gate
   - Tests first.
   - Use foreground HWND/PID/window rect evidence.

7. Vision/accessibility upgrade path
   - Design only after base click contracts are stable.
   - May introduce semantic verification separately from geometry verification.

## What Must Be Tested Before Implementation

- `browser_click` verified does not mean business outcome verified.
- `desktop_click` verified does not mean semantic button verified.
- selector uniqueness does not mean semantic correctness.
- deterministic command decomposition routes click intents correctly before dispatch.
- target resolution maps semantic phrases to browser target, desktop target, clarification, or approval.
- ambiguous click commands become `clarification_required` and do not dispatch.
- unresolved semantic targets do not fall back to blind coordinates.
- `unverified` always returns `success=False`.
- `approval_required` stays distinct from `unverified` and `failed`.
- dispatch exception returns `FAILED`, `success=False`.
- check-level evidence exists for every unverified result.
- execution evidence stores no screenshots, full page content, OCR text, or raw sensitive screen text.
- frontend/timeline must use backend-provided `verification_state`; no frontend inference.

## Scope Exclusions

Do not implement in this design sprint:

- Playwright DOM click evidence
- OS click evidence
- OCR
- vision
- accessibility tree
- autonomous retries
- workflow engine
- parser changes
- frontend changes
- replay/timeline changes
